## 软件设计原则

1. 单一职责原则 : 一个类做好只做一件事
2. 开闭原则: 过去的代码不去修改, 可以拓展以前的代码
3. 接口隔离原则: 一个接口只负责一类的工作
4. 里氏替换原则:  继承关系的代码不要修改父类的代码,通过新增子类代码来拓展功能
5. 依赖倒置原则: 定义成员变量, 返回值类型, 参数类型等不要使用具体的某个类,要使用接口 或者 抽象类
6. 迪米特法则: 两个不互相通信的类最好不要直接互相作用. 如果要调用另一个类的方法, 通过第三者转发


# 设计模式
结构类 , 行为类 , 创建类

## 单例模式
 * 懒汉式
 在调用getinstance才主动创建instance , 为了在多线程下保证单例和成功创建,使用双检查所和 volatile 来防止指令重拍
 * 饿汉式
 直接在成员变量上new 一个, 在类加载的初始化阶段就创建出instance对象

## 工厂模式
 定义一个接口, 新添加同类功能类的时候 新增实现类 

 创建工厂类 来创建具体这个接口的实现类实例

## 抽象工厂模式
 一个工厂只能生产一种产品, 一个抽象工厂可以生产多个产品族, 比如一个产品族可以有形状颜色大小 三种产品, 他的实现类工厂就是产品族的工厂

## 建造者模式
 当某个类的构造器参数过长或创建需要很多流程, 可以使用建造者模式, 这样就可以精细化创建过程, 有更好的扩展性,也能实现不可变对象(不用谢setter方法)

 角色有: director , Builder(interface) , BUilderImpl , Product 

## 原型模式
 用原型实例指定创建对象的种类, 通过拷贝方式来创建 实例

 拷贝方式:
* 浅克隆: 创建一个一模一样的对象,但是是一个引用数据类型,本质是一个指针,指向同一个内存空间
* 深克隆: 创建隐没一样的对象, 但是是在新的内存空间 . 使用序列化方式来实现最好, 把对象变成字节流,然后反序列化来

### 代码
``java
package com.zpero;
import java.io.ByteArrayInputStream;
import java.io.ByteArrayOutputStream;
import java.io.ObjectInputStream;
import java.io.ObjectOutputStream;
import java.io.OutputStream;
import java.io.Serializable;

public class TestProtoType {
    public static void main(String[] args) throws CloneNotSupportedException {
        Wife wife = new Wife(new Prama(12, 165));

        Wife clone1 = wife.clone();
        Wife clone2 = wife.clone();
        Wife clone3 = wife.clone();
        Wife clone4 = wife.clone();

        clone2.getPrama().setAge(18);
        System.out.println(clone1);
        System.out.println(clone2);
        System.out.println(clone3);
        System.out.println(clone4);
    }

}

class Prama implements Serializable{
    private static final long serialVersionUID = 1L;
    private int age;
    private int height;

    public Prama(int age, int height) {
        this.age = age;
        this.height = height;
    }

    public void setAge(int age) {
        this.age = age;
    }

    public int getAge() {
        return age;
    }

    public void setHeight(int height) {
        this.height = height;
    }

    public int getHeight() {
        return height;
    }

    @Override
    public String toString() {
        return "prama:{" + "height:" + this.height
                + ", age: " + this.age + "}";
    }

}

class Wife implements Cloneable, Serializable {

    private Prama prama;

    public Wife(Prama prama) {
        this.prama = prama;
    }

    public Prama getPrama() {
        return prama;
    }

    public void setPrama(Prama prama) {
        this.prama = prama;
    }

    @Override
    public String toString() {
        return "wife:{" + "prama:" + prama + "}";
    }

    @Override
    protected Wife clone() throws CloneNotSupportedException {
        // 深克隆
        // 1. 把对象写入字节流
        // 2. 从流里获取对象
        // 3. 返回对象

        Wife wife = (Wife) super.clone();
        try {
            ByteArrayOutputStream base = new ByteArrayOutputStream();
            ObjectOutputStream outputStream = new ObjectOutputStream(base);
            outputStream.writeObject(wife);

            ByteArrayInputStream byteArrayInputStream = new ByteArrayInputStream(base.toByteArray());
            ObjectInputStream input = new ObjectInputStream(byteArrayInputStream);

            Wife wife2 = (Wife) input.readObject();
            return wife2;
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

}
``

## 适配器模式
 将一个类的接口转换成客户希望的一个接口,使原本不兼容的类可以协同工作而不改变其内部结构
 
比如你要实现一个转换接口, 不仅可以播放mp3 , 也要播放mp4,vlc 
但是每种媒体的编码是不一样的, 我们有两种接口 : MP3 接口, vlc/mp4 接口
我们只调用一个接口的APi就完成所有媒体播放器的方法:

==加入适配器类== , 里面有两个接口成员, 同一个播放方法可以自己在方法里面判断并调用对应的API

## 桥接模式
 将抽象部分和实现部分分离, 使他们都可以独立地变化,

==抽取其中的一个维度, 让他成为独立的层次,把继承改为组合来实现这个问题==

比如 要表示图形, 图形有形状 和 颜色 , 要是创建绿色和红色的圆形,就要继承Shape
并创建两个类

不如把颜色抽取出来到另一个类里面, 把颜色作为图形的一个属性(成员变量)来变继承为组合

这样不管是创建红色绿色的圆形还是方形, 不再一个属性变动就要创建一个新的类了, 而是在相应的接口上进行组合, 只需要有形状和颜色接口就可以了

#### 适用场景: 一个类的变化是维度独立的,且这些维度都要拓展,不希望继承或多层次继承导致类的数量激增


> [!NOTE]
> 桥接模式避免了因为里面的一个类型的属性变化而导致的新增类,反而是把属性变接口, 一个接口可以对应多个实现类, 之后可以根据属性的不同而创建不同的类, 把类的数量从 m * n 降成 m+ n

## 组合模式
 把对象组合成树形结构表示整体部分的关系, 让用户对单个对象和多个对象的适用具有一致性

 对容器对象和 叶子对象进行统一操作, 且用户无需知道操作对象是容器还是叶子

## 装饰器模式
 
 向一个现有的对象添加新的功能, 同时又不改变他的结构, 动态的给一个对象添加一些额外的职责 

## 外观模式
  客户端访问系统的接口, 自动做很多个子系统的活儿

## 享元模式
创建大量对象会内存溢出, 把其中的共同部分抽象出来,有相同的业务需要就直接返回内存中这个已有的对象

享元工厂保存一个记录, 记录享元,每次访问从记录中找, 没有就创建按

## 代理模式
 用一个代理对象, 提供对这个对象访问的统一接口 

## 访问者模式
元素的执行算法可以随访问者的变化而变化 , 不同的访问者访问同一个 被访问者会 得到不同的结果

 用一个接口表示访问者, 每个访问者访问的方法内容不同 , 也就会得到不同的结果
